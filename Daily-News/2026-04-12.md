# OpenClaw 每日情報 2026-04-12

## 1. 今日重點摘要

1. **v2026.4.10 正式發布（4/11）**：本次版本帶來五大方向更新——捆綁式 Codex provider、可選 Active Memory 插件、實驗性 MLX 本地語音 Talk Mode、Microsoft Teams 訊息動作擴充，以及 QA 測試基礎設施強化（Matrix homeserver 即時通道），為 4 月份規模最大的單次版本。

2. **Active Memory Plugin 正式上線**：新增可選的 Active Memory 插件，在每次主要回覆前自動運行一個記憶子 Agent，從對話歷史中自動擷取相關偏好、上下文與過往細節。支援 `message`/`recent`/`full` 三種 context 模式，並可透過 `/verbose` 即時觀察、使用 prompt/thinking overrides 進行調校，是繼 v2026.4.9 REM 記憶回填後連續兩版的記憶體系強化。

3. **捆綁式 Codex Provider 與 GPT-5 系列正式整合**：`codex/gpt-*` 模型現由 Codex 自行管理 auth、原生 thread、模型發現與 compaction，與原本 `openai/gpt-*` 路徑分流。同時追加 `openai-codex/gpt-5.4-mini` forward-compat 支援及 GPT-5 prompt caching 穩定化修補。

4. **`openclaw exec-policy` 新 CLI 命令**：新增本機 `exec-policy` 命令，提供 `show`、`preset`、`set` 三個子命令，可同步 `tools.exec.*` 設定與本機 exec 批准檔案，解決過去 exec approval 設定分散難以管理的痛點。

5. **多平台修補：WhatsApp、Teams、Discord 同步修復**：WhatsApp 媒體附件處理恢復正常、跨重連的訊息路由維持穩定；Teams 恢復直接訊息、共用檔案的媒體下載，修復 SSO callback 及 Thread reply context 注入；Discord 升級 Carbon library 至 v0.15.0 並修復 TTS 語音筆記。

6. **安全強化持續推進（SSRF / Exec / 插件掃描）**：進一步收緊 SSRF 預設值、hostname allowlist 與 CDP discovery 防護；強化 exec preflight 讀取、host 環境 denylist 及插件相依套件掃描，延續 v2026.4.9 的安全修補主軸。

7. **GitHub Issue #37623：GPT-5.4 可設定但 runtime 不支援（Unknown model）**：用戶回報在 v2026.4.10 中設定 `openai-codex/gpt-5.4` 後，執行時出現 `Unknown model` 錯誤，model discovery 無法識別；相關追蹤 Issue #36817 仍持續開啟中，官方尚未給出時程。

8. **Reddit 社群討論：每次更新都踩坑**：r/openclaw 熱門貼文（97 讚）統計近期踩坑紀錄——v2026.3.28 重排 API keys、v2026.3.31 觸發 exec approval 無限迴圈、v2026.4.5 重命名 gateway entry 導致 provider 中斷、v2026.4.9 config validation 修改影響停用插件與 Telegram legacy keys。社群建議更新前務必備份 `~/.openclaw/config.yaml`。

9. **Anthropic OAuth 封鎖後的 Claude CLI 橋接方案仍是主流因應**：自 4/4 OAuth 限制生效後，`openclaw exec-policy` 新命令的上線被社群視為配合 Claude CLI 後端方案的配套工具，可更精細控制哪些 exec 行為委派給本機 `claude` binary，降低直接 API 呼叫需求。

## 2. LLM 串接實戰回報

### Claude（Anthropic）

- **OAuth 封鎖後遷移路徑確立**：4/4 生效的 OAuth 限制迫使大量用戶轉向獨立 API 金鑰或 Claude CLI 橋接方案。[Medium 教學](https://medium.com/@ulmeanuadrian/anthropic-just-cut-off-my-ai-agents-heres-how-i-fixed-it-in-20-minutes-741384d27080)說明 20 分鐘內完成遷移，核心步驟為：啟用 Extra Usage、設定 `ANTHROPIC_API_KEY`，或安裝 `claude` CLI 後將 OpenClaw provider 切換至 `claude-cli`。
- **Claude Opus 4.6 仍為預設 coding agent 首選**：有效 context window 200K tokens，tool use 穩定性高於其他 provider，但成本偏高；社群建議搭配 Claude Sonnet 4.6 處理中度複雜任務以降低費用。
- **Gateway restart 導致 Opus 費用爆炸（Bug #17589）**：Gateway 重啟時 in-flight fetch 被 AbortError 中斷，OpenClaw 誤判為需要完整重試，重送 100K+ token context 給 Opus，同時顯示誤導性「API rate limit reached」訊息。本次 v2026.4.10 包含部分修復，但完整解決方案仍在追蹤中。
- **來源**：[HN: Anthropic 封鎖 Claude Code 訂閱用於 OpenClaw](https://news.ycombinator.com/item?id=47633396)、[OpenClaw 說明頁](https://openclaw.rocks/blog/anthropic-oauth-ban)

### OpenAI / Codex（GPT-5 系列）

- **Codex Provider 整合踩坑**：v2026.4.10 捆綁 Codex provider 後，原本使用 `openai/gpt-5.4` 路徑的用戶需改寫 config 為 `openai-codex/gpt-5.4`，兩條路徑行為有差異（auth 流程、thread 管理均不同），舊有 workflow YAML 可能需要更新。
- **GPT-5.4 Unknown model 問題**：Issue #37623 確認 `openai-codex/gpt-5.4` 設定後 runtime 無法識別，目前建議改用 `openai-codex/gpt-5.4-mini` 作為暫時替代。
- **X.com 用戶 [@chris__sev](https://x.com/chris__sev/status/2024611917210538148) 心得**：「Codex 5.3 搭配 OpenClaw 表現相當不錯，比 Opus 4.6 少了一點個性但更穩定且便宜。」
- **來源**：[GPT-5.4 設定指南（Evolink）](https://evolink.ai/blog/openclaw-gpt-5-4-setup-guide)、[Issue #37623](https://github.com/openclaw/openclaw/issues/37623)

### Google Gemini

- **Per-model rate limit 觸發全 provider 冷卻（Issue #5744）**：Google 某單一模型達到 rate limit 後，OpenClaw 的 cooldown 邏輯會封鎖整個 Google provider，導致其他 Gemini 模型雖有剩餘 quota 也無法使用。本問題在 v2026.4.10 未見明確修復。
- **Gemini API Key Rotation 支援**：v2026.4.10 文件確認 Google provider 支援 optional API key rotation 及 `google/gemini-3.1-pro-preview`、`google/gemini-3-flash-preview` 等最新模型識別。
- **成本建議**：社群建議將簡單任務路由至 `gemini-3-flash-preview`，複雜任務留給 Claude 或 GPT-5，整體 API 費用可降低 40–60%。
- **來源**：[Issue #5744](https://github.com/openclaw/openclaw/issues/5744)、[OpenClaw 模型 Provider 文件](https://docs.openclaw.ai/concepts/model-providers)

### DeepSeek

- **DeepSeek V3.2 為社群最推薦的低成本選項**：$0.28/M input、$0.40/M output、164K context window，支援 function calling 與整合式 reasoning，透過 OpenRouter 串接後與 OpenClaw 工作流相容性佳。
- **透過 OpenRouter 橋接最穩定**：直接使用 DeepSeek 官方 API 在高負載下偶有 504 timeout；改用 OpenRouter 路由後穩定性明顯改善，但需考慮 OpenRouter 5.5% 加成費用。
- **來源**：[DeepSeek via OpenRouter 教學（Medium）](https://medium.com/@oo.kaymolly/connect-deepseek-to-openclaw-via-openrouter-7eb19ef61a84)、[haimaker.ai DeepSeek 比較](https://haimaker.ai/blog/best-deepseek-models-for-openclaw/)

### OpenRouter（多模型路由）

- **ClawRouter（BlockRunAI）**：新興 agent-native LLM router，支援 41+ 模型、< 1ms routing 決策，並整合 USDC 支付（Base & Solana via x402），定位為 OpenClaw 專用的智慧路由層。
- **路由策略建議**：簡單任務（檔案讀取、短回答）→ `deepseek/deepseek-v3.2` 或 `google/gemini-3-flash-preview`；複雜推理 → `claude-opus-4-6` 或 `openai/gpt-5`；費用最佳化可節省 40–60%。
- **來源**：[6 best LLM routers for OpenClaw（DEV Community）](https://dev.to/sophiaashi/6-best-llm-routers-for-openclaw-in-2026-17oa)、[ClawRouter GitHub](https://github.com/BlockRunAI/ClawRouter)

### Grok（xAI）

- **1.8M Context Window 為最大賣點**：透過 OpenRouter 串接 Grok 模型，context window 達 1.8M tokens，multimodal 支援，tool calling 表現穩定，適合需要超長上下文的 agent 工作流。
- **目前透過 OpenRouter 路由為主流方式**，尚無原生 OpenClaw Grok provider。

## 4. 值得追蹤的後續議題

1. **GPT-5.4 Runtime 支援完整化（Issue #37623 & #36817）**：`openai-codex/gpt-5.4` 目前仍顯示 Unknown model，官方追蹤 Issue 開啟中，預計下一版修復——需持續關注是否影響使用 Codex provider 的現有 agent workflow。

2. **Active Memory Plugin 穩定化與隱私疑慮**：Active Memory 在 v2026.4.10 中為可選插件，但社群已開始討論記憶資料的本地儲存位置、是否加密、是否會跨 session 混入不相干記憶等問題，相關文件尚不完整。

3. **Anthropic Extra Usage 費用異常監控**：OAuth 封鎖後大量用戶轉至 Extra Usage 按量付費，已有回報指出未預期的費用激增（部分源於 Bug #17589 的 AbortError 重試問題），建議設定 Anthropic 帳戶的每日消費上限並啟用告警。

4. **Google Gemini Per-model Rate Limit 觸發全 Provider 冷卻（Issue #5744）**：v2026.4.10 未見修復，重度使用 Gemini 多模型架構的用戶需持續關注，考慮透過 OpenRouter 路由以繞過此問題。

5. **`openclaw exec-policy` 命令與安全管控落地**：新 CLI 命令上線後，企業用戶如何整合進 CI/CD pipeline 並設定最小權限 exec policy，預計將成為下一階段的社群討論重點，值得追蹤官方文件更新。

6. **ClawRouter 與 OpenRouter 競合格局**：BlockRunAI 的 ClawRouter 以 agent-native 路由切入，加入 Web3 支付（USDC），與現有 OpenRouter 形成差異化競爭。其 41+ 模型覆蓋與 < 1ms routing 的實際效能表現值得後續驗證。
